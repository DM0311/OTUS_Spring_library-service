🎯 Цель проекта
Разработать RESTful-сервис управления библиотекой, который позволяет:

Пользователям искать книги, бронировать их, оставлять отзывы.

Администраторам управлять каталогом книг, пользователями и просматривать аудит действий.

Реализовать JWT-аутентификацию (в будущем) для разграничения доступа.

🧩 Функциональные блоки
1. Управление каталогом
   Книги – CRUD, поиск с фильтрацией, рейтинг, управление экземплярами (totalCopies / availableCopies).

Авторы – CRUD (связь многие-ко-многим с книгами).

Жанры – CRUD (связь многие-ко-многим с книгами).

2. Пользователи и безопасность
   Регистрация (пока без email-подтверждения).

JWT-аутентификация (Access + Refresh токены).

Управление пользователями (блокировка, назначение ролей).

Штрафы (penaltyPoints) и автоматическая блокировка при превышении лимита.

3. Бронирование (аренда) книг
   Создание бронирования (проверка доступности).

Продление срока (макс. 1 раз).

Возврат книги (обновление availableCopies).

Отмена бронирования.

Автоматическая проверка просрочек (Scheduler) → начисление штрафов.

История бронирований для пользователя.

4. Отзывы
   Оставление отзыва на книгу (только если пользователь брал книгу? опционально).

Редактирование/удаление своего отзыва.

Средний рейтинг книги обновляется автоматически.

5. Уведомления
   Системные уведомления для пользователей:

О просрочке возврата.

О начислении штрафов.

О доступности забронированной книги.

Просмотр и отметка о прочтении.

6. Аудит (AuditLog)
   Логирование всех изменяющих действий (CRUD, бронирования, блокировки и т.д.).

Реализован через событийную модель (ApplicationEventPublisher + @EventListener).

Просмотр логов доступен только администратору.

7. Статистика (для администратора)
   Топ-10 популярных книг (по количеству бронирований или рейтингу).

Активные читатели (кто чаще всего берёт книги).

Количество просроченных бронирований.

🔄 Бизнес-процессы (основные сценарии)
Регистрация пользователя → создание записи в users и назначение роли USER.

Авторизация (JWT) → выдача Access/Refresh токенов.

Поиск книг → фильтрация по названию, автору, жанру, году.

Бронирование книги:

Проверка availableCopies > 0.

Уменьшение availableCopies на 1.

Создание записи в bookings со статусом ACTIVE.

Логирование события BOOKING_CREATE.

Возврат книги:

Увеличение availableCopies на 1.

Обновление статуса бронирования на RETURNED.

Логирование BOOKING_RETURN.

Продление бронирования:

Проверка, что продление ещё не использовалось (extensionCount < 1).

Обновление dueDate (+14 дней).

Логирование BOOKING_EXTEND.

Просрочка (Scheduler):

Ежедневная проверка dueDate < CURRENT_DATE.

Начисление penaltyPoints (1 балл в день).

Блокировка пользователя при penaltyPoints >= 10.

Логирование BOOKING_OVERDUE, PENALTY_APPLIED, USER_AUTO_BLOCKED.

Оставление отзыва:

Проверка, что пользователь брал книгу (опционально).

Обновление среднего рейтинга книги.

Логирование REVIEW_CREATE.

Администрирование:

Блокировка/разблокировка пользователей.

Назначение ролей.

CRUD книг, авторов, жанров.

Просмотр статистики и аудит-логов.

🔧 Ключевые технические решения
Аудит через события – вместо явных вызовов AuditService в контроллерах мы публикуем AuditEvent и обрабатываем его в AuditEventListener.

Асинхронность – логирование выполняется в отдельном потоке (не блокирует основной ответ).

Scheduler – проверка просрочек выполняется автоматически каждый день в 00:00.

Flyway – все изменения БД версионируются.

MapStruct – маппинг Entity ↔ DTO (меньше ручного кода).

Hibernate – Lazy-загрузка + EntityGraph для оптимизации запросов.

📌 Ключевые моменты для документации
BookingStatus: PENDING, ACTIVE, RETURNED, OVERDUE, CANCELLED.

ActionType – полный список см. в пакете enums.

Правила штрафов: +1 балл за день просрочки, блокировка при 10+ баллах.

Продление: доступно только 1 раз (независимо от статуса).
